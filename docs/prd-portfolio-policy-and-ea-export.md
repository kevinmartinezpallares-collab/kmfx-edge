# KMFX Edge PRD: Portfolio Policy and EA Export

Status: product requirements document
Last updated: 2026-05-14
Purpose: define how portfolios in KMFX Edge should evolve from grouped views into executable policy objects and exportable EA packages.

## Summary

A portfolio in KMFX should become more than a reporting layer.

It should become a policy object that defines how a cluster of accounts may be used.

From that policy, KMFX should eventually be able to export or sync an EA package that enforces risk, eligibility, and allocation rules inside MetaTrader.

## Product thesis

Most traders with multiple accounts do not need another dashboard card.

They need a way to express:

- which accounts belong together
- what each account is for
- what risk budget each receives
- which strategies may run where
- what should happen when the same idea appears across several accounts

That is the role of a portfolio policy.

## Product goal

Make multi-account trading manageable through policy.

The product should help traders:

- allocate risk intentionally
- avoid duplicated correlated exposure
- decide where a setup belongs
- enforce those decisions consistently

## In-scope

- portfolio definition
- account clustering
- risk budget by account
- strategy permissions
- symbol permissions
- concentration caps
- session permissions
- copy/split/block rules
- exportable policy package
- EA enforcement concept

## Out of scope for V1

- autonomous alpha generation
- no-code strategy builder
- ML-driven strategy discovery
- direct broker abstraction layer

## Primary users

- trader operating several personal and funded accounts
- trader separating accounts by strategy or role
- trader wanting one system to enforce allocation discipline

## Core concepts

### Portfolio

A named cluster of accounts with shared operating policy.

### Account role

Examples:

- lead account
- follower account
- challenge account
- payout-protection account
- experimental account

### Strategy permission

Defines whether a strategy or setup is:

- allowed
- limited
- blocked

For a given account or portfolio.

### Allocation policy

Defines:

- target risk budget
- max capital share
- max heat
- max concurrent trades
- duplication rules across accounts

### Export package

A machine-readable policy artifact generated by KMFX for EA use.

## Functional requirements

## 1. Portfolio creation

The system must allow the user to create a portfolio with:

- name
- description
- objective
- included accounts
- account roles

## 2. Portfolio policy definition

The system must allow rules such as:

- max risk per account
- max total heat
- max exposure by symbol
- max exposure by factor/currency
- max allocation by strategy
- max concurrent positions
- allowed sessions
- allowed instruments

## 3. Strategy-account mapping

The system must allow the user to define:

- which strategies may trade which accounts
- whether a strategy is primary or secondary for an account
- whether a strategy is blocked after a threshold is reached

## 4. Cross-account routing rules

The system must support decisions such as:

- copy this setup to all eligible accounts
- split this setup across selected accounts
- send only to the safest account
- block setup because portfolio concentration is too high

## 5. Portfolio status view

The user must be able to see:

- current portfolio heat
- concentration risk
- account contribution
- remaining budget
- blocked conditions

## 6. Export package generation

The system must be able to generate a policy package that contains:

- portfolio id
- account identifiers
- account roles
- allowed symbols
- strategy permissions
- risk caps
- session windows
- concentration caps
- routing rules
- emergency freeze rules

## 7. EA enforcement model

The system must support at least the concept of an EA that:

- reads the package
- validates whether a trade is eligible
- sizes it according to policy
- blocks or routes it according to rules
- returns audit telemetry if implemented

## Recommended export models

### A. Risk Guardian EA

Purpose:

- enforce trade eligibility and lot sizing only

### B. Portfolio Router EA

Purpose:

- decide where a valid setup should be placed across accounts

### C. Strategy-bound EA

Purpose:

- combine execution strategy with KMFX policy

Recommended order:

- A first
- B second
- C later

## Key user flows

### Flow 1. Build a portfolio

1. user creates a portfolio
2. assigns accounts
3. sets account roles
4. sets risk budgets and strategy permissions
5. reviews summary

### Flow 2. Validate a setup

1. user has a strategy/setup idea
2. KMFX checks eligible accounts
3. KMFX shows whether the setup should be copied, split, or blocked
4. user acts or exports

### Flow 3. Export policy package

1. user selects a portfolio
2. user chooses export model
3. KMFX validates data completeness
4. KMFX generates package
5. EA reads package and enforces policy

## Technical product notes

Official MQL5 documentation suggests a few practical constraints:

- EAs are event-driven and chart-attached.
- Multi-symbol logic is possible but needs careful design.
- `WebRequest()` is synchronous.
- file access is sandboxed.

So the best product framing is:

- KMFX defines policy
- EA consumes a lightweight policy package
- EA stays thin and deterministic

Not:

- giant opaque export blobs
- constant heavy remote dependency

## UX principles

- a portfolio should read like an operating mandate
- rules must be human-readable
- blocked states should be explicit
- export should feel safe and versioned
- comparison should privilege decision usefulness over decoration

## Risks

- overcomplication if every edge case enters V1
- user confusion if routing rules are too abstract
- false trust if export is sold as autonomous intelligence

## Phasing

### Phase 1

- portfolio objects
- account roles
- risk budgets
- strategy permissions
- portfolio detail view

### Phase 2

- routing recommendations
- copy/split/block logic
- concentration policies

### Phase 3

- export package format
- Risk Guardian EA support
- audit telemetry loop

### Phase 4

- Portfolio Router EA
- richer factor-aware routing
- optional strategy-bound modules

## Open questions for later implementation

- whether export should be file-based, sync-based, or both
- how strategy identifiers should be normalized
- whether routing should start rule-based only or include score-based ranking
